# AI 变更风险审查平台：初始产品文档说明

## 1. 文档目的

本文档用于说明“AI 变更风险审查平台”课题与产品的一体化设计思路，作为项目初始阶段的统一说明文档。本文档的主要用途有三点：

1. 帮助研发人员、评审人员与项目参与者快速理解本课题的研究背景、研究目标与产品落地方向。
2. 作为项目实现阶段的基础说明，指导后续架构设计、模块拆分、数据模型设计与 MVP 实现。
3. 作为 Codex 等智能编码代理的优先阅读文档，使其能够在统一目标、统一约束和统一边界下协助完成设计与开发。

本文档强调“从课题研究到产品落地”的连续性，不将本项目定义为一个单纯的 AI 插件，而是定义为一个可接入研发流程、可配置、可扩展、可沉淀组织经验的 AI 变更风险审查平台。

---

## 2. 课题背景与问题定义

### 2.1 背景

随着 AI 编码助手、Agent 开发模式与代码生成能力的快速普及，开发阶段的代码生产方式正在发生明显变化。开发者越来越频繁地使用 AI 辅助写代码、解释代码、生成测试、修复问题和理解项目结构。在这种背景下，传统意义上的 Code Review 价值边界也在发生变化。

一方面，AI 已经能够较好地处理部分低层次问题，例如基础语法问题、部分空指针风险、常见规范问题、简单重复代码问题等。另一方面，在真实研发场景中，影响系统稳定性和上线质量的核心问题，往往并不只是“代码是否能运行”，而是：

- 设计是否合理
- 改动是否影响核心链路
- 是否存在隐藏的一致性风险
- 是否引入并发问题、性能问题或兼容性问题
- 是否缺少灰度、监控、回滚、告警等上线保障措施
- 是否触达数据库、缓存、MQ、配置中心等关键资源

因此，单纯做一个“在 IDE 中对代码聊天式 review”的工具，很容易与已有 AI 插件重复，缺少明确创新价值与组织落地价值。

### 2.2 当前研发中的真实痛点

结合现实研发流程，团队通常已经具备或部分具备以下能力：

- 开发者在本地开发阶段使用 AI 编码工具
- GitLab / GitHub 平台中存在自动化 AI Review 能力或基础流水线检查
- 上线前仍然需要组织人工 Code Review 或发布前检查

这说明问题并不在于“缺一个入口”，而在于缺少一套真正面向研发流程、面向上线前风险、面向组织知识沉淀的审查机制。

### 2.3 核心问题定义

本课题与产品要解决的核心问题是：

**如何围绕代码变更本身，在研发关键节点自动识别影响面与潜在风险，形成结构化风险卡片，并结合团队历史审查经验持续沉淀组织级风险知识，从而提升人工 Review 的聚焦度和上线前问题发现能力。**

---

## 3. 课题研究方向

### 3.1 研究主题

本课题拟聚焦以下研究主题：

**基于代码变更影响分析的上线前风险审查机制研究与平台实现。**

该研究方向不再将目标限定为“重新做一个 AI Code Review 插件”，而是转向以下几个更有价值的维度：

1. 从“代码内容检查”转向“变更风险识别”
2. 从“单点工具能力”转向“研发流程接入能力”
3. 从“即时输出建议”转向“组织知识沉淀”
4. 从“通用智能审查”转向“团队可配置的专项审查模板”

### 3.2 研究目标

本课题的研究目标包括：

1. 建立一套基于代码变更的影响分析方法，识别本次改动涉及的核心资源与高风险区域。
2. 建立一套面向研发关键节点的风险审查策略，在 Merge Request、分支合并、构建或发布前自动触发审查。
3. 建立结构化风险卡片输出机制，将 AI 审查结果转化为便于人工决策的信息载体。
4. 建立 Code Review 结果沉淀机制，将人工 review 经验、高频隐患、常见规范与历史问题持续转化为组织知识资产。
5. 构建可配置的审查模板机制，支持后端、前端和通用研发场景的差异化审查。

### 3.3 预期创新点

本课题预期的创新点主要体现在以下方面：

#### 创新点一：从代码审查转向变更风险审查
不是泛化地分析代码质量，而是围绕变更范围、影响资源与上线风险做审查，更贴近真实研发流程。

#### 创新点二：将审查结果结构化为风险卡片
不是输出大段笼统文本，而是形成结构化、可推送、可复核、可沉淀的风险卡片。

#### 创新点三：引入组织级经验沉淀机制
将多人 Code Review 结果、历史问题模式、团队规范、事故经验逐步沉淀为可复用规则和知识模板。

#### 创新点四：面向多技术栈的模板化审查能力
以模板为抽象单元，实现前端、后端、通用场景的定制化审查，而非只服务某个 IDE 或某个语言环境。

## 4. 产品定位

### 4.1 产品名称

暂定产品名称：

**AI 变更风险审查平台**

### 4.2 产品定位

本产品定位为：

**一个接入研发流程、支持多团队配置、能够围绕代码变更自动生成风险审查结果并持续沉淀组织经验的研发质量平台。**

它不是单纯的 AI 聊天工具，也不是只服务于某个 IDE 的插件，而是一个平台型产品。

### 4.3 产品目标用户

目标用户包括：

- 后端开发工程师
- 前端开发工程师
- Reviewer
- 测试工程师
- 技术负责人 / 小组长
- 发布负责人
- 架构师 / 质量治理人员

### 4.4 产品核心价值

本产品希望带来的核心价值包括：

1. 在研发关键节点自动识别变更风险，减少人工 review 的盲审。
2. 帮助 reviewer 快速聚焦高价值问题，提高 Code Review 效率与有效性。
3. 降低上线前风险遗漏概率，提升研发质量保障能力。
4. 将团队的审查经验从零散评论升级为可沉淀、可配置、可复用的知识资产。
5. 为不同团队、不同技术栈提供统一平台和差异化模板支持。

---

## 5. 产品边界与非目标

### 5.1 产品边界

本产品当前聚焦于“变更驱动的风险审查”。主要关注以下内容：

- 代码变更分析
- 风险识别
- 审查结果生成
- 风险卡片展示
- 流程节点触发
- 审查记录沉淀
- 规则与模板配置

### 5.2 非目标

为了避免目标发散，以下内容不作为一期核心目标：

- 不以 IDEA 插件为主产品形态
- 不追求完全替代人工 Code Review
- 不在一期内做复杂全链路静态分析引擎
- 不在一期内做大而全的个人画像系统
- 不在一期内覆盖所有语言和所有业务框架
- 不依赖单一大模型输出作为唯一判断依据

---

## 6. 核心产品思路

### 6.1 总体思路

围绕“代码变更”这一核心输入，在研发流程的关键节点自动触发审查，形成如下闭环：

1. 获取变更信息
2. 分析变更影响面
3. 基于规则和 AI 识别风险
4. 生成结构化风险卡片
5. 在 MR、IM、发布平台等位置进行触达
6. 人工 review 后将结果回流，沉淀为可复用知识

### 6.2 核心产品原则

本产品遵循以下原则：

- 平台优先，而非插件优先
- 流程接入优先，而非单点交互优先
- 结构化输出优先，而非大段自然语言优先
- 配置化与模板化优先，而非硬编码优先
- 规则与 AI 结合，而非纯 AI 黑盒判断
- 先做 MVP 闭环，再逐步增强智能深度

## 7. 产品核心能力设计

### 7.1 变更影响分析能力

该能力是产品的核心基础能力。系统需要根据 Merge Request、commit 范围、分支差异等输入，识别本次改动涉及的关键资源。

一期重点关注以下资源类型：

- 接口
- 数据库表 / SQL / Mapper
- 缓存
- MQ 生产与消费逻辑
- 配置中心 / nacos / yml / properties

目标不是一次性做出非常复杂的代码理解能力，而是先建立一套可扩展的变更分类与影响识别框架。

### 7.2 风险识别能力

在获得变更分析结果后，系统根据内置规则、团队规则与 AI 增强分析，识别潜在风险项，例如：

- 接口兼容性风险
- 数据一致性风险
- 幂等与重复消费风险
- 缓存一致性风险
- 配置变更风险
- 性能与并发风险
- 监控与告警缺失风险
- 灰度与回滚准备不足风险

### 7.3 风险卡片能力

风险卡片是产品的核心输出形式。它应当具备以下特点：

- 结构化
- 简洁
- 可推送
- 可阅读
- 可追溯
- 可回流

风险卡片建议包含：

- 本次改动摘要
- 影响范围
- 风险等级
- 风险项列表
- 推荐检查项
- 推荐关注角色
- 是否建议专项 review
- 历史相似问题提示（后续增强）

### 7.4 审查记录沉淀能力

系统需要记录每次审查过程与结果，为后续分析和知识沉淀提供基础。沉淀的对象包括：

- 审查任务记录
- 风险卡片结果
- 人工 review 反馈
- 采纳建议与忽略建议
- 高频问题类型
- 历史改进案例

### 7.5 模板与规则配置能力

为了让产品能够服务不同研发团队，系统需要支持模板化配置。例如：

- backend-default
- frontend-default
- general-default

模板能够决定：

- 启用哪些规则
- 重点关注哪些变更类型
- 推荐哪些检查项
- 采用什么推送策略

## 8. 产品形态设计

### 8.1 总体产品形态

本产品采用“一个平台 + 多个入口”的形态，而不是“一个插件就是产品”。

### 8.2 平台端

平台端是核心，负责：

- 项目接入管理
- 审查任务管理
- 模板与规则管理
- 审查结果展示
- 风险卡片查看
- 审查记录沉淀
- 数据统计分析
- 权限与配置管理

### 8.3 使用入口

建议初期至少支持以下三个入口：

#### 入口一：GitLab MR 审查入口
在 MR 创建或更新时触发自动审查。

#### 入口二：Jenkins / 构建 / 发布前入口
在构建成功、提测或发布申请前触发风险分析。

#### 入口三：Web 平台手动发起入口
开发者、架构师或测试可手动指定 MR、分支差异或 commit 范围，发起专项审查。

### 8.4 可选附属入口

后续可扩展支持：

- IDE 插件入口
- 命令行入口
- 发布平台入口
- 企业 IM 机器人命令入口

这些入口均不是产品本体，而是平台能力的触达方式。

## 9. MVP 范围设计

### 9.1 一期 MVP 目标

一期 MVP 的目标是打通一条完整闭环：

**GitLab MR webhook -> 变更分析 -> 风险识别 -> 风险卡片 -> 钉钉推送 -> 审查结果落库 -> Web 查看**

### 9.2 一期 MVP 功能清单

一期建议实现以下功能：

1. GitLab Merge Request webhook 接入
2. 审查任务创建与落库
3. changed files 与 diff 基础解析
4. API / DB / CACHE / MQ / CONFIG 五类变更识别
5. 默认风险规则引擎
6. 风险卡片生成
7. 钉钉 webhook 推送
8. 审查任务列表与详情页
9. 模板能力的最小实现（至少支持 backend-default）
10. 审查记录数据结构落地

### 9.3 一期 MVP 暂不实现内容

- 高精度深层语义代码理解
- 全项目调用图分析
- 历史问题自动聚类分析
- 多模型动态路由
- 复杂权限体系
- 插件端复杂交互

## 10. 典型使用场景

### 场景一：MR 自动风险审查

开发者提交或更新 Merge Request 后，系统自动触发审查，生成风险卡片并推送给提交人和 reviewer。reviewer 可据此优先关注高风险区域。

### 场景二：发布前风险确认

在 Jenkins 构建成功或发布申请阶段，系统再次触发审查，根据本次变更与已有规则生成发布前风险提示，辅助发布负责人判断是否需要补充验证。

### 场景三：专项人工发起审查

开发者或架构师在 Web 平台上手动指定一个 MR 或 commit 范围，选择模板后触发专项审查，用于高风险需求、核心模块改造或发布前复核。

---

## 11. 多团队与多技术栈扩展思路

### 11.1 初期从后端切入

考虑到后端场景中更容易体现数据库、缓存、MQ、配置中心等资源风险，一期优先面向后端落地。

### 11.2 中期扩展到前端

后续可扩展前端场景模板，重点关注：

- API 调用兼容性
- 状态管理副作用
- 路由与权限变更
- 埋点完整性
- UI 回归风险
- 错误处理与降级能力
- 渲染性能问题

### 11.3 平台化扩展策略

通过“模板 + 规则 + 知识库”的方式，把能力做成平台，而不是绑定语言和框架。这样不同团队可在统一产品下做差异化配置。

---

## 12. 数据沉淀与组织价值

本产品不仅输出即时结果，更重要的是沉淀组织级研发质量资产。长期看，系统可以逐步形成以下沉淀：

- 高频风险模式库
- 团队代码规范库
- Review 检查项库
- 历史问题案例库
- 组件使用约束库
- 不同业务线模板库

这意味着产品的长期价值不在于“某一次 AI 审查结果”，而在于持续帮助团队把经验显性化、结构化、平台化。

---

## 13. 成功指标与效果评估

为了验证产品的真实价值，后续需要建立效果评估指标。建议包括：

- 审查任务触发次数
- 风险卡片查看率
- 风险建议采纳率
- 高频问题覆盖率
- 人工 Review 聚焦度提升情况
- 发布前问题发现数量
- 漏检问题回流数量
- 团队规范沉淀数量

一期阶段可先关注以下基础指标：

- 审查链路是否可用
- 风险卡片是否可读
- reviewer 是否愿意使用
- 是否能识别若干典型后端风险

## 14. 产品实施路径建议

### 阶段一：明确文档与架构

完成产品文档、架构设计、数据模型、风险卡片 schema、MVP 范围收敛。

### 阶段二：实现最小闭环

打通 GitLab webhook、变更分析、风险卡片、钉钉推送、记录落库与 Web 查看。

### 阶段三：增强模板与规则

引入后端模板、规则配置能力和基础管理页面。

### 阶段四：沉淀知识与复盘机制

将人工 Review 结果回流，并逐步建立规范库、案例库和风险模式库。

### 阶段五：扩展前端与多团队使用

增加前端模板与通用模板能力，使产品从“后端课题验证”走向“团队平台能力”。

---

## 15. 对 Codex 的实现指导说明

为了让 Codex 更有效理解并参与实现，本项目建议遵循以下实现原则：

1. 先文档后代码
2. 先实现 MVP 闭环，再做增强能力
3. 先规则化，再逐步接入 AI 增强
4. 所有输出都必须结构化
5. 所有模块必须边界清晰、便于扩展
6. 不要将产品能力绑定单一入口
7. 所有实现应以平台化与模板化为长期方向

Codex 在实现过程中，应优先围绕以下对象建模：

- Project
- ReviewTask
- ChangeAnalysisResult
- RiskItem
- RiskCard
- ReviewTemplate
- NotificationRecord

在实现顺序上，应优先完成：

1. 文档与 schema
2. 后端基础骨架
3. webhook 接入与落库
4. 变更分析器
5. 风险规则引擎
6. 风险卡片生成
7. 钉钉推送
8. Web 页面展示

---

## 16. 总结

本项目并不是要再做一个同质化的 AI Code Review 插件，而是希望围绕真实研发流程，构建一套面向上线前风险识别、面向组织经验沉淀、面向多团队可配置使用的 AI 变更风险审查平台。

从课题研究角度看，它研究的是“如何基于代码变更建立一套有效的上线前风险审查机制”；从产品落地角度看，它最终形成的是“一个可接入流程、可自动触发、可持续沉淀知识的研发质量平台”。

这也是本课题相对单纯插件实现更具有研究价值、工程价值与推广价值的根本原因。
